# INSTALL.md — the Ralph Harness install playbook (single source of truth)

> **You are the installer.** You are an AI coding agent (opencode, or anything
> with shell and filesystem tools) and the user asked you to install the Ralph
> Harness into the project from whose root you are running. The Harness is an
> autonomous-development **runtime** (two loops, an event bus, on-disk CoALA
> memory, and a Presentation Layer), **not** a product to be built.
>
> **This file is the single source of truth for the install.** Follow it end to
> end, in order — do not skip phases. This instruction **deliberately contains no
> information about the target project**: you discover it by reconnaissance
> (Phase 0) and validate every step against what is on disk. Stop at any gate and
> ask before a destructive action. The install must be **idempotent**. The hard
> rules at the bottom apply across every phase.

---

## Concept: three classes of file

- **FIXED** — copied as-is, identical in any project: `template/.harness/` (the
  whole runtime), `template/.opencode/`, `template/opencode.json`. On conflict,
  **merge**, do not overwrite.
- **SEED** — generated by reading the project, because it describes *this*
  project: `AGENTS.md` (project root), `.harness/specs/PRODUCT.md`,
  `.harness/specs/ARCHITECTURE.md`. The moulds are in `template/_seed/`.
- **ADAPTABLE** — when the project **already has** the file (typically
  `AGENTS.md`): merge, preserving the owner's content and adding the Harness
  sections in marked blocks.

Mental rule: *same in any project → FIXED; describes this project → SEED; already
existed and the Harness also needs it → ADAPTABLE.*

---

## Phase 0 — Reconnaissance (DO NOT SKIP)

Discover the terrain before touching anything (you have not cloned yet — that is
Phase 1). Record the answers; they drive every later phase.

1. **New or existing project?** Is the root empty or does it already have code?
2. **Git?** Is there a `.git/`? Current branch? Uncommitted changes? (If so, warn
   and ask before proceeding.)
3. **Already has the Harness?** Do `.harness/`, `.opencode/`, `AGENTS.md` exist?
   If so, this is an **update** — act by merge.
4. **Stack.** Detect language and manager: `package.json` (Node — check for
   `pnpm-workspace.yaml`/`turbo.json`), `pyproject.toml`/`requirements.txt`
   (Python), `Cargo.toml` (Rust), `go.mod` (Go), etc.
5. **Real commands.** Extract the true build/test/lint/type-check commands from
   the manifests. You will need them in `AGENTS.md` and the gates.
6. **Conventions and boundaries.** Note generated directories that must never be
   touched (`dist/`, `node_modules/`, `.next/`, `target/`).

Present a summary before installing anything (Phase 1 onward). Cloning (Phase 1)
is harmless scaffolding; the first changes to the project happen in Phase 2.

## Phase 1 — Clone the template

Clone this template into an isolated temporary directory **outside** the
project's tree, depth 1:

```bash
git clone --depth 1 https://github.com/pobruno/opencode-harness-loop .harness-tmp
```

Everything you copy from here on comes from `.harness-tmp/template/`. The clone is
scaffolding — Phase 6 deletes it.

---

## Phase 2 — Install the FIXED files (idempotent, with merge)

Copy into the project root, **creating what is missing and merging what exists**:

- `.harness-tmp/template/.harness/`   → `.harness/`   (the whole runtime, incl.
  the header files for `specs/ROADMAP.md`, `specs/SPRINT.md`, `inbox/`,
  `memory/`, and the `presentation/` layer)
- `.harness-tmp/template/.opencode/`  → `.opencode/`  (the 7 agents + commands)
- `.harness-tmp/template/opencode.json`→ `opencode.json` (if present, **merge**
  the `mcp`, `permission`, `skills`, and `instructions` keys; never clobber the
  owner's `default_agent`)

Conflict rules:
- If a directory already exists, copy **only the missing files**. Do not overwrite
  an existing file without asking.
- The Harness's `build` agent intentionally overrides opencode's built-in of the
  same name — call this out so the owner understands.
- Ensure scripts are executable:
  `chmod +x .harness/*.sh .harness/runtime/*.sh .harness/presentation/html/*.sh`.

The template already ships the empty state/event/log/tasks directories (with
`.gitkeep`). Do not delete them.

## Phase 3 — Generate the SEED files (by reading the project)

### 3a. AGENTS.md — GENERATE, do not copy

Use `template/_seed/AGENTS.seed.md` as the skeleton and fill every
`{{PLACEHOLDER}}` from Phase 0. Keep the `[Harness]` sections verbatim (working
context, subagent policy, completion markers, the 6-agent contracts, the decision
doctrine, git-safety). `AGENTS.md` must be minimal and high-signal — only what the
agent cannot infer from the code. Keep it under ~32 KiB.

**If an `AGENTS.md` already exists (ADAPTABLE):** preserve all the owner's content
and add the `[Harness]` sections under marked headings. Note any conflict in the
final report.

### 3b. .harness/specs/PRODUCT.md and .harness/specs/ARCHITECTURE.md

- **Existing project:** populate from what you detected — the real modules,
  stack, and visible contracts (read via the `explore` subagent). Do not invent.
- **New project:** copy the moulds from `template/_seed/` with the questions the
  bootstrap will answer. Do not fill with assumptions.

`PRODUCT.md` is mandatory — the runtime refuses to start without it. In
`ARCHITECTURE.md`, keep the **INVARIANTS** section from the seed and either fill it
now (if you can infer the non-negotiable rules) or leave it for `/bootstrap` — the
invariants are the only absolute veto the autonomous loop cannot override, so they
matter. `.harness/specs/DECISION_POLICY.md` ships with default WSJF weights; leave
it for the human to tune at bootstrap.

### 3c. ROADMAP.md and SPRINT.md

Already shipped with headers in `.harness/specs/`. Leave them empty — the
`plan` agent fills them.

## Phase 4 — Configure the MCPs

Read `.harness-tmp/template/.harness/mcp/manifest.json`. For each server,
register it in the project's `opencode.json` `mcp` block. Rules:

- **Credentials are never invented.** For each token/key, write a placeholder
  (`{env:ENV_VAR_NAME}`), add the variable to `.env.example` with a comment, and
  record the pending item.
- Remote OAuth MCPs need no manual token — just the URL.
- Do not duplicate a server already configured.
- **Do not enable servers marked `on_demand`** (e.g. the headless browser,
  synthesized later by the `mcp-synth` skill). Leave them noted.

---

## Phase 5 — Validate

Confirm the installation is coherent, without a full agent run:

1. **Counters / loop decision:** `bash .harness/runtime/state.sh counters` and
   `bash .harness/runtime/state.sh next_loop` read the documents without error.
2. **Decision Engine:** `bash .harness/runtime/decide.sh wsjf 5 3 2 4` prints a
   number and `bash .harness/runtime/decide.sh door "drop a column"` prints
   `one-way` — the autonomous decision helpers work.
3. **Runtime dry-run:** `bash .harness/loop.sh --once --no-intake` should reach
   `idle`/`DONE` (empty docs) without error — it will not call the model when
   there is no work.
3. **Dashboard:** `bash .harness/dashboard.sh` generates
   `.harness/presentation/html/index.html`.
4. **State:** `jq . .harness/state/runtime.json` is valid JSON.
5. **Gates:** if a toolchain is installed, `bash .harness/runtime/gates.sh --list`
   shows the detected gates. If auto-detection misses the real commands, write
   `.harness/gates.local.sh` exporting `GATE_TYPECHECK`, `GATE_LINT`,
   `GATE_TEST`, `GATE_BUILD`.
6. **TUI (optional):** if Python is present, note that
   `pip install -r .harness/presentation/textual/requirements.txt` enables the
   Textual TUI. Do not install it unless asked.

Do not force anything to pass — confirm the pieces come up.

---

## Phase 6 — Clean up

1. Delete the clone: `rm -rf .harness-tmp`.
2. Adjust `.gitignore` for the ephemeral artefacts that do **not** go to git:
   `.harness/state/`, `.harness/events/`, `.harness/logs/`,
   `.harness/presentation/html/index.html`, and any real `.env`. Keep versioned:
   `.harness/specs/`, `.harness/memory/` (long-term memory is part of the
   project), `.harness/runtime/`, `.harness/presentation/`, `.harness/skills/`,
   `AGENTS.md`, `.opencode/`, `opencode.json`.
3. **Do not** commit automatically — let the user review the diff.

---

## Phase 7 — Final report

Deliver a clear summary:

- **Copied (FIXED):** what was installed from the runtime.
- **Generated (SEED):** the `AGENTS.md` and specs you wrote, with one sentence on
  what you detected.
- **Merged (ADAPTABLE):** if an `AGENTS.md` pre-existed, what you preserved/added
  and any conflict kept from the owner.
- **Human pending items:** credentials to fill in `.env` (one line per MCP), and
  any open decision.
- **Next step:** run `/bootstrap` (or paste the bootstrap prompt) to populate the
  specs, the ARCHITECTURE invariants, and the decision policy (the WSJF weights),
  then `bash .harness/loop.sh` for the first cycle — or open the Textual TUI. Use
  `opencode --agent desk` to dump work into the inbox.

Confirm `.harness-tmp` was removed and the project is self-sufficient.

---

## Hard rules across all phases

- **Idempotent always.** Copy only what is missing; merge, never blind-overwrite;
  detect "already installed" and skip.
- **Never overwrite the owner's files** beyond the marked `[Harness]` blocks. On a
  collision, stop and ask.
- **Never invent secrets.** Placeholder + `.env.example` + a pending-item note.
- **Never force-push, rewrite `.git` history, or commit on the user's behalf.**
- **Stop on the first hard failure.** Surface the exact output; do not auto-retry.
- **If you don't know, ask.** Never guess a project root, a stack, or a path.
- **The clone is scaffolding.** When you finish, only the installed files remain.
